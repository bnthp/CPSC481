G1 Calendar

High-fidelity prototype and heuristic evaluation

About the Prototype

When you're trying to figure out a time that all your friends are free so you can plan a meetup, it can be difficult and annoying to get all their schedules and manually figure out the best time and day. Team One’s aim is to design a scheduling application called G1 Calendar that visually displays times people are “busy” so you can easily find out when people aren’t busy. You can then mark the schedule with that event so everyone in the group can see.

The G1 Calendar application will be an Android mobile app. Our high-fidelity prototype for G1 Calendar was created using atomic.io.

The link to our high-fidelity prototype demonstrated in class: G1 Calendar hi-fi prototype


Heuristic Evaluation

The heuristic evaluation of the G1 Calendar high-fidelity prototype inspected the ten usability heuristics for user interface design. The ten usability heuristics developed by Jakob Nielsen are the following:

  1. Visibility of system status
  2. Match between system and the real world
  3. User control and freedom
  4. Consistency and standards
  5. Error prevention
  6. Recognition rather than recall
  7. Flexibility and efficiency of use
  8. Aesthetic and minimalist design
  9. Help users recognize, diagnose, and recover from errors
  10. Help and documentation

Each heuristic listed is examined in this evaluation. The evaluation was performed by Matthew and Christopher.

1. Visibility of system status

Currently the visibility of system status is minimal. During account creation, it would be beneficial to have a prompt advise the user the account was created successfully rather than just being put back on the login page. Additionally there is no highlighting of clickable buttons to indicate they are intractable, but this is partially offset by the low number of non-clickable text/buttons. One positive is that it is easy to see if the system is in group mode or personal mode by visual cues.

2. Match between system and the real world

The text used for input prompts and descriptions are clear and easy to understand as they use very natural language. Account login and creation are both set up in the standardized way and match most other real world systems. Icons for the most part are well chosen with the exception of a pencil for “title” in events and the choice of a “file save” icon to save the event which are not terrible, but could be more intuitive.

3. User control and freedom

Given the low number of actions and screens, there are very few situations which should require backing out. The prototype only has one occurrence of a back button, and that is on the account creation page. When creating an group or personal event, if the user taps the close button by mistake, an error prevention window is prompted. The system asks if the user actually wants to cancel creating an event.

One improvement to the system would be to allow undo/redo of the last event added, in case the user incorrectly deletes an event that they actually wanted to keep.

4. Consistency and standards

When cycling between views or opening the menu, only a portion of the screen changes with the rest remaining the same. This consistency between system states makes it feel grounded and easy to navigate as you almost always still see a portion of the screen you recognize. Standardized images were also selected to complement entry fields and always use symbols like “+” to denote to “add” a friend/group/event.

5. Error prevention

Given the simplicity of the system there are only a few key places where error prevention is needed. It was implemented successfully during account creation by having it ask to retype the password to ensure no slips by the user. Less successful error prevention was present when canceling an event creation. Although it does produce a confirmation prompt when canceling, the prompt is poorly worded and hard to determine which button will cancel the event and which will allow continued editing.

5. Recognition rather than recall

As the app mostly revolves around one screen, little recollection is required. The group/personal calendar state is viewable in the top right, and the busy-ness of a day is recognizable by how darkly shaded the day is on the calendar. Both these features can be easily be recognized instead of opening other dialog windows to check. The only view that does requires recall is the event creation as it fill the entire screen preventing viewing of the calendar but requires information from the calendar to know when people are available.

7. Flexibility and efficiency of use

No accelerators have been implemented but this is mostly because of the simplicity of the app. Most activities require only a few taps so there is very little to optimize for an experienced user. Two places that could be improved through is the login page and the weekly calendar view. The login page could use a “remember username/password” option, and the weekly calendar view should have the ability to select a time in the weekly view, and have that selected time set as the event start time.

8. Aesthetic and minimalist design

In this section, we will provide some tasks and their descriptions. These tasks will be implemented to our scheduling application.

Aesthetically, the app is decent but could use improvements for the text styles. In the “add to group” prompt, there are 5 styles of font on one window with various colours, sizes, and some bolded producing an incohesive look. For minimalistically, the app is very successful. There are few buttons/options per page and each is separated to help prevent the feeling of them being cluttered.

9. Help users recognize, diagnose, and recover from errors

Similar to error prevention, errors can only really occur during the account pages and creating an event. In the account page, entering invalid credentials produces an error message indicating bad credentials and a reset password link is provided just below the password prompt. Unfortunately, once the wrong password text appears once, it does not flash or otherwise indicate a change if you try again with an incorrect password to indicate that is still the problem. There is also no error checking in event creation to notify users that they have selected an end time before the start time.

10. Help and documentation

No help documentation is currently provided but all options and fields are clearly labeled and have icons for easy comprehension. Despite this, the app could use a help page that describes the purpose of the app and the basic use flow to help a new user get started faster.
